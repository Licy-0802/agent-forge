# 完整工作流和交接规范

> 本 skill 的核心变化：每个项目的**开发角色**和**测试角色**不再是固定一个，而是由主理人立项时通过角色分析动态决定数量和分工。

---

## 工作流总览

```
项目立项 → 角色分析 → 需求阶段 → 开发阶段 → 测试阶段 → 项目闭环
   ↑                    ↑            ↑          ↑           ↑
  主理人             产需(1个)    开发(N个)   测试(M个)    主理人
```

### 阶段说明

| 阶段 | 参与者 | 说明 |
|------|--------|------|
| 立项 + 角色分析 | 主理人 | 确定项目类型，分析需要几个开发/测试实例 |
| 需求阶段 | 产需(1个) | 产出需求文档 |
| 开发阶段 | 开发(N个) | N 个开发实例并行或串行工作 |
| 测试阶段 | 测试(M个) | M 个测试实例独立验证 |
| 项目闭环 | 主理人 | 审核所有测试报告，确认交付 |

### 阶段闸门规则

一个项目只有在**当前阶段的所有角色实例都完成**时，才能推进到下一阶段：

- `需求中` → `开发中`：产需角色完成 ✅
- `开发中` → `测试中`：**所有开发实例完成** ✅
- `测试中` → `已完成`：**所有测试实例完成** ✅

### 每个角色的生命周期

```
                        ┌───→ 🚫 blocked (阻塞)
                        │
⏳ pending (待启动) ──→ ▶️ active (进行中) ──→ ✅ done (已完成)
```

## 阶段一：项目立项 + 角色分析（主理人）

### 触发条件
有新的产品需求或业务需求需要启动项目。

### 步骤
1. **收集项目信息**：项目名称、业务目标、期望交付、约束
2. **评估优先级**：P0 / P1 / P2 / P3
3. **确定项目范围**：In Scope / Out of Scope
4. **角色分析**（关键新增步骤）：
   a. 选择项目类型（单应用 / 全栈 / 多服务 / 多端）
   b. 决定开发实例数量和分工（如：2个 = 前端开发 + 后端开发）
   c. 决定测试实例数量和分工（如：1个 = 集成测试）
   d. 决定工作模式（并行 / 串行 / 混合）
5. **创建项目**：运行 `scripts/init_project.py`
6. **指派产需角色**：`scripts/transition_project.py PROJ-0001 --role 产需 --start`

### 角色分析决策树

```
项目需要几个独立的任务流？
├── 1个任务流（单应用）→ 1个开发实例
├── 2个任务流
│   ├── 前后端独立 → 前端开发 + 后端开发（并行）
│   └── 有依赖 → 核心开发 + 扩展开发（串行）
├── 3个任务流 → Web + API + Admin 或 服务A + 服务B + 服务C
└── 更多 → 按模块拆分

测试策略：
├── 所有模块一起交付 → 1个集成测试实例
├── 各模块独立交付 → 每个模块1个测试实例
└── 混合 → 模块测试 + 集成测试
```

## 阶段二：需求阶段（产需）

### 入口条件
- 产需角色已启动（`--role 产需 --start`）
- 已有产品文档或业务背景说明

### 步骤
1. 阅读并理解产品文档
2. 编写 PRD
3. 拆分为用户故事（每个故事独立可交付）
4. 为每个用户故事编写验收标准（Given/When/Then）
5. 交付主理人审核

### 退出条件
- 产需角色标记完成：`scripts/transition_project.py PROJ-0001 --role 产需 --done`

---

## 阶段三：开发阶段（开发 N 个实例）

### 入口条件
- 项目已进入 `development` 阶段
- 主理人已启动所有开发实例

```bash
scripts/transition_project.py PROJ-0001 --role 前端开发 --start
scripts/transition_project.py PROJ-0001 --role 后端开发 --start
```

### 工作模式

**并行模式**（默认）：
- 所有开发实例同时启动
- 每个实例独立完成自己的模块
- 互不等待

**串行模式**：
- 实例按顺序启动
- 实例 A 完成后，启动实例 B
- 适合有强依赖的场景（如 后端完成 → 前端开发）

**混合模式**：
- 部分实例并行，部分串行
- 例如：前端和后端并行 → 联调开发在两者之后

### 实例工作流程
1. 理解需求文档和验收标准
2. 设计技术方案（如需）
3. 按用户故事实现
4. 每个故事完成后自测（验收标准验证 + 边界 + 异常）
5. 标记完成：`scripts/transition_project.py PROJ-0001 --role 标签 --done`

### 退出条件
- 所有开发实例标记为 `done`
- 无 P0/P1 级别的已知问题

### 常见场景

#### 场景1：全栈项目，前后端并行
```
主理人 → [启动 前端开发] ──→ 前端编码 ──→ ✅ done
       → [启动 后端开发] ──→ 后端编码 ──→ ✅ done
                                      ↓
                             主理人确认: 两个都完成
                                      ↓
                                 进入测试阶段
```

#### 场景2：多微服务，串行
```
主理人 → [启动 核心服务开发] ──→ 完成 ──→ ✅ done
                                      ↓
                           [启动 扩展服务开发] ──→ 完成 ──→ ✅ done
                                                         ↓
                                                 进入测试阶段
```

#### 场景3：动态追加角色
```
主理人发现需要一个额外的"数据迁移"开发角色：

1. 手动编辑 projects.json，在 roles 数组中追加：
   {"label": "数据迁移", "type": "dev", "status": "pending", "note": "MySQL→PG"}

2. 启动该角色：
   scripts/transition_project.py PROJ-0001 --role 数据迁移 --start
```

## 阶段四：测试阶段（测试 M 个实例）

### 入口条件
- 项目已进入 `testing` 阶段
- 主理人已启动所有测试实例

```bash
scripts/transition_project.py PROJ-0001 --role 集成测试 --start
scripts/transition_project.py PROJ-0001 --role 性能测试 --start
```

### 步骤
1. 阅读需求文档和验收标准
2. 编写测试用例（功能流程 + 边界 + 异常）
3. 执行测试
4. 记录 Bug（标注等级）
5. 输出测试报告
6. 标记完成：`scripts/transition_project.py PROJ-0001 --role 标签 --done`

### 退出条件
- 所有测试实例标记为 `done`
- 无 P0 缺陷，P1 已修复并回归
- P2/P3 在测试报告中记录，主理人决定是否修复

---

## 阶段五：项目闭环（主理人）

### 入口条件
- 所有测试实例已完成

### 步骤
1. 审核测试报告
2. 确认所有 P0/P1 已修复
3. 确认 P2/P3 有处理计划
4. 确认交付物完整
5. 推进项目到完成：
   ```bash
   scripts/transition_project.py PROJ-0001 --advance
   ```

## 多项目并行管理

### 项目队列管理

主理人维护项目队列。每个项目在任何时刻只处于一个阶段，但一个项目的开发阶段可能包含多个并行的角色实例。

```
项目队列示例：
[P0] 用户中心重构
     类型: 全栈 | 模式: 并行
     ▶️ 前端开发 (dev) | ▶️ 后端开发 (dev) | ⏳ 集成测试 (qa) | ⏳ 产需 (pm) ✅

[P1] 支付模块升级
     类型: 单应用
     ▶️ 开发 (dev) | ⏳ 测试 (qa) | ✅ 产需 (pm)

[P2] 数据看板 V2
     类型: 多服务
     ▶️ 服务A开发 | ▶️ 服务B开发 | ⏳ 服务C开发 | ⏳ 集成测试
```

### 阶段串行化（跨项目）

每个角色类型（产需、开发、测试）**同时只能处理一个项目的主线任务**：
- 产需在处理项目 A，项目 B 的需求阶段必须等产需释放
- 开发在处理项目 A，项目 B 的开发阶段等开发释放

### 角色实例的跨项目分配

当一个开发实例（比如"前端开发"）在处理项目 A 时，他不能同时处理项目 B 的前端开发。但如果项目 A 有**不同的角色实例**（比如"后端开发"），则后端开发在项目 A 空闲时可以被分配到项目 B。

### 例外规则
- P0 紧急项目可以抢占正在进行的 P2/P3 项目资源
- 被抢占的项目状态变为"阻塞"，挂起等待
- 主理人决定是否将被抢占项目的部分角色实例临时调给 P0 项目

### 每日站会模板

主理人每日检查：
1. 每个项目的当前阶段
2. 每个角色实例的状态（⏳待启动 / ▶️进行中 / ✅已完成 / 🚫阻塞）
3. 有没有阻塞项
4. 有没有新项目需要进入队列
5. 资源分配是否需要调整

## 迁移说明：从旧模型升级

如果你之前使用过旧的"固定四角色"版本，正在运行的项目需要通过以下方式迁移：

1. 打开 `~/.codex/projects/projects.json`
2. 将每个项目的 `owner` 字段替换为 `roles` 数组：

```json
// 旧格式
{
  "owner": {"requirements": "已指派", "development": "已指派", "testing": ""}
}

// 新格式
{
  "roles": [
    {"label": "产需", "type": "pm", "status": "done", "note": ""},
    {"label": "开发", "type": "dev", "status": "active", "note": ""},
    {"label": "测试", "type": "qa", "status": "pending", "note": ""}
  ]
}
```
