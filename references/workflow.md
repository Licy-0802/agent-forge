 # 完整工作流和交接规范

 ## 工作流总览

 ```
 项目立项 → 需求阶段 → 开发阶段 → 测试阶段 → 项目闭环
    ↑           ↑           ↑           ↑           ↑
   主理人       产需        开发        测试         主理人
 ```

 ## 阶段一：项目立项（主理人）

 ### 触发条件
 有新的产品需求或业务需求需要启动项目。

 ### 步骤
 1. 收集项目信息：项目名称、业务目标、期望交付时间、已知约束
 2. 评估项目优先级：P0（紧急）/ P1（高）/ P2（中）/ P3（低）
 3. 确定项目范围：明确做什么、不做什么
 4. 创建项目卡片（见项目模板）
 5. 注册项目到看板
 6. 指派给产需角色

 ### 输出
 - 项目卡片（Project Card）
 - 项目进入看板，状态变为"需求中"

 ## 阶段二：需求阶段（产需）

 ### 入口条件
 - 已收到主理人指派的项目卡片
 - 已有产品文档或业务背景说明

 ### 步骤
 1. 阅读并理解产品文档，如有不明之处向主理人确认
 2. 编写 PRD（产品需求文档）
 3. 将 PRD 拆分为用户故事（每个故事独立可交付）
 4. 为每个用户故事编写验收标准（Given/When/Then）
 5. 输出数据字典和业务规则说明（如涉及）
 6. 将需求文档交付主理人审核

 ### 评审检查点
 - 需求是否覆盖了所有产品文档中的要点？
 - 每个用户故事是否都有清晰的验收标准？
 - 开发是否能根据需求估算工作量？
 - 测试是否能根据验收标准编写测试用例？

 ### 输出
 - PRD + 用户故事列表（含验收标准）
 - 状态更新为"待主理人审核"

 ## 阶段三：开发阶段（开发）

 ### 入口条件
 - 已收到主理人确认的已审核需求文档

 ### 步骤
 1. 理解需求文档和验收标准，如有模糊之处立即问主理人
 2. 设计技术方案（如需）
 3. 按用户故事逐个实现（建议按依赖关系排序）
 4. 每个用户故事完成后进行自测：
    a. 验收标准逐条验证
    b. 边界情况测试
    c. 异常流程测试
 5. 自测通过后通知主理人
 6. 所有用户故事完成后提交整体完成报告

 ### 退出条件
 - 所有用户故事编码完成
 - 所有自测通过
 - 无 P0/P1 级别的已知问题

 ### 输出
 - 功能代码
 - 自测报告
 - P0/P1 问题清零确认

 ## 阶段四：测试阶段（测试）

 ### 入口条件
 - 已收到主理人分配的可测试项目
 - 已有需求文档和验收标准作为测试依据

 ### 步骤
 1. 阅读需求文档和验收标准
 2. 编写测试用例（覆盖功能流程、边界、异常）
 3. 执行测试
 4. 记录 Bug（每个 Bug 标注等级）
 5. 输出测试报告
 6. 给出验收结论

 ### 退出条件
 - 所有测试用例执行完毕
 - 无 P0 缺陷
 - P1 缺陷已全部修复并回归通过
 - P2/P3 缺陷在测试报告中记录，主理人决定是否修复

 ### 输出
 - 测试用例文档
 - Bug 列表（分等级）
 - 测试总结报告
 - 验收结论

 ## 阶段五：项目闭环（主理人）

 ### 入口条件
 - 测试验收结论为"通过"或"有条件通过"

 ### 步骤
 1. 审核测试报告
 2. 确认所有 P0/P1 缺陷已修复
 3. 确认 P2/P3 缺陷已有处理计划
 4. 确认交付物完整
 5. 归档项目文档
 6. 在看板中标记项目为"已完成"

 ### 输出
 - 项目总结
 - 项目归档

 ## 多项目并行管理规则

 ### 项目队列管理

 主理人维护一个统一的**项目队列**。每个项目在任何时刻只处于一个阶段。

 ```
 项目队列示例：
 [P0] 用户中心重构 → 开发阶段（开发中）
 [P1] 支付模块升级 → 需求阶段（产需进行中）
 [P1] 数据看板 V2   → 测试阶段（测试进行中）
 [P2] 消息推送系统   → 待启动（排队中）
 [P3] 后台 UI 优化   → 待启动（排队中）
 ```

 ### 阶段串行化

 每个角色（产需、开发、测试）**同时只能处理一个项目的主线任务**。

 - 如果产需正在处理项目 A，项目 B 的产需阶段必须排队等待
 - 如果开发正在处理项目 A，项目 B 的必须等开发释放
 - 这样虽然看起来慢，但避免了上下文切换导致的效率损失

 ### 例外规则

 - P0 紧急项目可以抢占正在进行的 P2/P3 项目资源
 - 抢占后，被抢占项目状态变为"阻塞"，挂起等待
 - 主理人需安抚被抢占项目的干系人

 ### 每日站会模板

 主理人每日检查以下内容：
 1. 每个项目当前状态是什么？
 2. 每个项目今天的目标是什么？
 3. 有没有阻塞项需要主理人处理？
 4. 有没有新的项目需要进入队列？
 5. 资源分配是否需要调整？

 ```
 - [列出功能点]

 - [列出明确不做的事项]

 [待启动 / 需求中 / 开发中 / 测试中 / 已完成 / 阻塞中]

 - 产需：[姓名]
 - 开发：[姓名]
 - 测试：[姓名]

 - [依赖项目或外部系统]
 ```
