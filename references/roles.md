# 角色职责详解

> 本 skill 采用**角色类型 + 角色实例**模型。以下首先定义四种角色类型，然后给出主理人做角色分析时的决策指南。

---

## 角色类型概览

| 类型 | 标识 | 是否可多实例 | 说明 |
|------|------|------------|------|
| 主理人 | `principal` | 否（唯一） | 项目经理/决策者，不实例化 |
| 产需 | `pm` | 否（每项目1个） | 将产品构想转化为需求文档 |
| 开发 | `dev` | **是**（可多个） | 按需求实现功能 |
| 测试 | `qa` | **是**（可多个） | 按需求验证质量 |

---

## 1. 主理人 (Principal) — 唯一

主理人是整个多项目管理体系的中心决策者。**不实例化**，由使用系统的人（或 Codex 扮演）担任。

### 核心职责
- **项目立项**：明确项目目标、范围、交付物和截止时间
- **角色分析**：分析项目结构，确定需要哪些角色实例（见下文"角色分析指南"）
- **优先级决策**：为每个项目分配 P0-P3 优先级
- **资源分配**：将角色实例指派给具体执行者
- **进度跟踪**：维护项目看板，通过 `--role --done` 跟踪每个角色的完成进度
- **冲突仲裁**：当多个项目争抢同一角色实例时做出裁决
- **质量把关**：确认所有测试实例完成，决定是否达到交付标准

### 决策原则
- 业务价值优先于技术完美
- 高优先级项目拥有资源优先权
- P0 > P1 > P2 > P3
- 一个项目只有**全部开发实例完成**才能进入测试阶段
- 一个项目只有**全部测试实例完成**才能闭环

### 角色分析指南

这是主理人最核心的决策工作。立项时，回答以下问题：

#### 第一步：判断项目类型

| 项目类型 | 典型特征 | 典型角色结构 |
|---------|---------|-------------|
| **单应用** | 单一后端或单一前端，一个人能覆盖所有开发 | 1 产需 + 1 开发 + 1 测试 |
| **全栈** | 前后端在同一个项目，需要独立分工 | 1 产需 + 前端开发 + 后端开发 + 集成测试 |
| **多服务** | 多个后端微服务在同一个项目中迭代 | 1 产需 + N 个服务开发 + 1 集成测试 |
| **多端** | Web/Mobile/Admin 等多端同时开发 | 1 产需 + Web开发 + Mobile开发 + ... + 集成测试 |

#### 第二步：决定开发实例数量

不是"需要几个人"——而是"需要几个可独立评估进度的任务流"：

- 一个前端开发和一个后端开发是否可以**独立工作**？
  - 是 → 两个开发实例，并行
  - 否（前端依赖后端API）→ 两个开发实例，串行（后端先启动）
- 多个微服务之间如何？
  - 无依赖 → 并行
  - 服务A 依赖 服务B 的接口 → 可以并行但需要先定义契约

#### 第三步：决定测试实例

- 是否需要按**模块/端**独立测试？→ 每个模块一个测试实例
- 是否只需要一个**整体集成测试**？→ 一个测试实例

#### 第四步：决定工作模式

- **并行**：所有开发实例同时启动，互不等待。适合模块独立、依赖明确的场景
- **串行**：开发实例按顺序启动，前者完成后者开始。适合强依赖场景
- **混合**：部分并行、部分串行。例如前端和后端并行，但移动端等后端完成再启动

## 2. 产需 (PM) — 每项目一个

### 核心职责
- 需求分析：理解业务目标和用户痛点
- PRD编写：包含功能描述、业务规则、数据定义
- 用户故事拆分：将大需求拆分为独立可交付的故事
- 验收标准定义：每个用户故事编写 Given/When/Then 验收标准
- 需求评审：组织主理人和开发测试共同评审

### 输出物
- PRD + 用户故事列表（含验收标准）
- 业务规则说明
- 数据字典
- 界面原型说明（如需）

### 质量要求
- 每个用户故事必须有量化验收标准
- 需求粒度：一个开发可在 1-3 天完成
- 尽量避免"系统应该..."这类模糊表述

---

## 3. 开发 (Developer) — 可多实例

每个开发实例独立负责一个模块/端的实现。

### 核心职责
- 技术方案设计
- 功能实现（针对自己的模块）
- 自测：完成编码后按验收标准验证
- 进度汇报：主动向主理人汇报进度和阻塞
- 注意：开发实例之间**不直接交互**，任何协作需求通过主理人协调

### 实例标签建议

| 实例标签 | 类型 | 典型负责 | 建议专注领域 |
|---------|------|---------|-------------|
| 前端开发 | dev | 前端 UI + 交互 | React/Vue/Next.js |
| 后端开发 | dev | API + 数据 | Go/Node/Python |
| 服务A开发 | dev | 微服务A | 具体技术栈 |
| Web开发 | dev | Web端 | React/Next.js |
| Mobile开发 | dev | 移动端 | Flutter/RN |

### 工作原则
- 只针对已确认的需求文档进行开发
- 需求不清晰时暂停并反馈主理人
- 完成一个用户故事后通过 `--role 标签 --done` 更新
- 阻塞持续超过 4 小时立即上报

### 输出物
- 功能代码（含单元测试）
- 自测报告
- 部署说明（如需）

---

## 4. 测试 (QA) — 可多实例

每个测试实例独立负责一个模块/端的质量验证。

### 核心职责
- 测试用例设计：功能测试、边界测试、异常场景
- 功能验证：执行测试用例
- 缺陷报告：记录 Bug（含等级）
- 回归测试：开发修复后进行回归
- 测试报告：输出总结报告

### 缺陷等级
- **P0-致命**：系统崩溃、核心功能不可用、数据丢失
- **P1-严重**：主要功能异常、流程中断
- **P2-一般**：次要功能异常、边界未处理
- **P3-轻微**：文案拼写、样式偏移

### 实例标签建议

| 实例标签 | 类型 | 典型负责 |
|---------|------|---------|
| 集成测试 | qa | 全链路功能验证 |
| 功能测试 | qa | 模块级功能测试 |
| 性能测试 | qa | 负载/压力测试 |

### 输出物
- 测试用例文档
- Bug 列表（含等级）
- 测试总结报告
- 验收结论（通过/有条件通过/不通过）

---

## 角色间协作规范

| 交接方 | 接收方 | 交接内容 | 验收标准 |
|--------|--------|----------|----------|
| 主理人 | 产需 | 项目卡片 + 产品文档 | 产需理解项目范围 |
| 产需 | 主理人 | PRD + 用户故事 | 主理人审核通过 |
| 主理人 | 开发实例N | 已审核需求文档 | 开发理解需求和验收标准 |
| 开发实例N | 主理人 | 完成代码 + 自测报告 | 功能完整，自测通过 |
| 主理人 | 测试实例M | 需求文档 + 可测试环境 | 测试理解测试范围 |
| 测试实例M | 主理人 | 测试报告 + 验收结论 | 测试完成，结果明确 |

**关键规则**：所有角色间交接必须通过主理人。角色实例之间不直接交接。
