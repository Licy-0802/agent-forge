 # 角色职责详解

 ## 1. 主理人 (Principal)

 主理人是整个多项目管理体系的中心决策者，负责项目从立项到闭环的全生命周期管理。

 ### 核心职责
 - **项目立项**：明确项目目标、范围、交付物和截止时间
 - **优先级决策**：为每个项目分配 P0-P3 优先级，决定哪个项目先进入哪个阶段
 - **资源分配**：将项目指派的对应角色（产需→开发→测试），确保每个阶段有明确负责人
 - **进度跟踪**：维护项目看板，定期检查各项目进展
 - **冲突仲裁**：当多个项目争抢同一资源时，做出裁决
 - **质量把关**：确认测试结果，决定项目是否达到交付标准
 - **风险管控**：识别并记录项目风险，制定应对措施

 ### 决策原则
 - 业务价值优先于技术完美
 - 高优先级项目拥有资源优先权
 - 当资源冲突时，P0 > P1 > P2 > P3
 - 项目须阶段流转，不可跳级

 ### 输出物
 - 项目卡片（含范围、优先级、估算工时）
 - 项目看板（当前所有项目的状态总览）
 - 阶段流转指令（将项目从一个角色移交给另一个角色）

 ## 2. 产品转需求 (Product-to-Requirements)

 负责将模糊的产品构想转化为清晰、可执行的需求文档，是衔接想法和实现的关键桥梁。

 ### 核心职责
 - **需求分析**：理解产品文档中的业务目标和用户痛点
 - **需求文档编写**：撰写 PRD（产品需求文档），包含功能描述、业务规则、数据定义
 - **用户故事拆分**：将大需求拆分为可独立交付的用户故事
 - **验收标准定义**：为每个用户故事编写明确的验收标准（Given/When/Then 格式）
 - **需求评审**：组织主理人和开发、测试共同评审需求，确保各方理解一致
 - **需求变更管理**：当需求发生变化时，更新文档并通知相关方

 ### 输出物
 - PRD（产品需求文档）
 - 用户故事列表（含验收标准）
 - 业务规则说明
 - 数据字典（如涉及数据结构定义）
 - 界面原型或交互说明（如需要）

 ### 质量要求
 - 每个用户故事必须有可量化的验收标准
 - 需求描述必须无歧义，避免"系统应该..."这类模糊表述
 - 需求的粒度：一个开发可在1-3天内完成

 ## 3. 开发 (Developer)

 根据需求文档实现功能，是代码产出的核心执行者。

 ### 核心职责
 - **技术方案设计**：针对需求文档设计技术实现方案
 - **功能实现**：编写代码完成用户故事
 - **自测**：完成编码后进行基本的单元测试和功能自测
 - **进度汇报**：主动向主理人汇报完成进度和遇到的阻塞
 - **代码审查**：参与代码审查（如果团队有其他开发人员）

 ### 输出物
 - 技术方案说明（可选，适用于复杂功能）
 - 功能代码（含单元测试）
 - 完成自测报告
 - 部署说明或迁移脚本（如有需要）

 ### 工作原则
 - 只针对已确认的需求文档进行开发
 - 遇到需求不清晰时，暂停并反馈给主理人，而非自行假设
 - 完成一个用户故事后立即更新状态
 - 阻塞持续超过 4 小时立即上报

 ## 4. 测试 (Tester)

 负责根据需求文档验证实现的质量，确保交付物符合验收标准。

 ### 核心职责
 - **测试计划编写**：根据用户故事和验收标准制定测试计划
 - **测试用例设计**：编写功能测试、边界测试、异常场景测试用例
 - **功能验证**：执行测试用例，验证功能是否满足需求
 - **缺陷报告**：记录 Bug（含复现步骤、期望结果、实际结果、严重等级）
 - **回归测试**：开发修复 Bug 后进行回归验证
 - **测试报告**：输出测试总结报告，包含通过率、未修复 Bug 列表、风险评估

 ### 缺陷等级定义
 - **P0-致命**：系统崩溃、核心功能不可用、数据丢失
 - **P1-严重**：主要功能异常、流程中断、性能严重下降
 - **P2-一般**：次要功能异常、UI 显示异常、边界情况未处理
 - **P3-轻微**：文案拼写、样式偏移、体验优化建议

 ### 输出物
 - 测试用例文档
 - Bug 列表（含等级和状态）
 - 测试总结报告（通过率、未修复清单、风险评估）
 - 验收结论（通过 / 有条件通过 / 不通过）

 ## 角色间协作规范

 | 交接方 | 接收方 | 交接内容 | 验收标准 |
 |--------|--------|----------|----------|
 | 主理人 | 产需 | 项目卡片 + 产品文档 | 产需理解项目范围 |
 | 产需 | 主理人 | PRD + 用户故事 | 主理人审核通过 |
 | 主理人 | 开发 | 已审核的需求文档 | 开发理解需求和验收标准 |
 | 开发 | 主理人 | 完成代码 + 自测报告 | 功能完整、自测通过 |
 | 主理人 | 测试 | 需求文档 + 可测试环境 | 测试理解测试范围 |
 | 测试 | 主理人 | 测试报告 + 验收结论 | 测试完成、结果明确 |

 **关键规则**：所有角色间交接必须通过主理人。产需、开发、测试之间不直接交接，确保主理人对项目全局有完整掌控。
